# Order Book（訂單簿）

**訂單簿**是金融交易撮合系統用來記錄所有未成交買賣請求的帳本。市場上每一個尚未撮合的限價單（limit order）都會以「價格 + 數量」的形式登記在簿上，等待對手方出現。

# 結構

訂單簿分為兩側：
- **買方 (Bids)**：所有想買入的限價單，依價格**由高到低**排序。
- **賣方 (Asks / Offers)**：所有想賣出的限價單，依價格**由低到高**排序。

兩側最接近的一筆價格稱為：
- **Best Bid**：目前買方願意出的最高價。
- **Best Ask**：目前賣方願意收的最低價。
- **Spread = Best Ask − Best Bid**：兩者之差，反映流動性與交易成本。

範例（ETH/USDC）：

| 賣方 (Asks) | 數量 |
| --- | --- |
| 2,005 | 5 |
| 2,002 | 3 |
| **2,001** | 2 ← Best Ask |
| ↕ Spread = 2 | |
| **1,999** | 4 ← Best Bid |
| 1,998 | 6 |
| 1,995 | 10 |
| 買方 (Bids) | 數量 |

# 撮合機制

當新訂單進來時，撮合引擎依下列規則處理：
- **市價單 (Market Order)**：吃掉對面最優價的掛單，直到數量滿足。
- **限價單 (Limit Order)**：若價格能與對手方匹配（買價 ≥ Best Ask 或賣價 ≤ Best Bid），立即成交；否則登記到簿上等待。

撮合的優先順序遵循 **Price-Time Priority**：
1. **價格優先**：較好的價格（買方更高、賣方更低）優先成交。
2. **時間優先**：同價格時，先掛的單先成交。

# 兩種實作架構

訂單簿可建構在伺服器上（傳統 CEX），或建構在區塊鏈上（鏈上 DEX）。完整對比見 [[Week 12 Note]]，這裡聚焦於**訂單簿層面**的差異：

| 維度 | 伺服器訂單簿 | 鏈上訂單簿 |
| --- | --- | --- |
| 撮合速度 | 微秒級 | 受區塊時間限制（秒～分鐘） |
| 取消訂單 | 免費、即時 | 需付 gas fee |
| 抗審查 | ✗ 交易所可凍結帳戶 | ✓ 任何人皆可掛單 |
| Front-running 來源 | 交易所內部 | 礦工 / MEV searcher |
| 透明度 | 後台不公開 | 鏈上完全公開 |
| 系統穩健性 | 單點故障風險 | 由區塊鏈共識保障 |

# 鏈上訂單簿的挑戰

鏈上訂單簿的可行性受限於下列因素：
- **每筆掛單/取消都要 gas**：高頻造市成本極高。
- **MEV (Maximal Extractable Value)**：礦工可重排或搶先交易，掛單者容易被狙擊。
- **延遲與吞吐量**：以太坊 L1 每秒只能處理十幾筆交易，遠不及 CEX。

因此早期 DEX 多數選擇用 [[Week 12 Note|AMM]] 取代訂單簿。近年來在高吞吐 L2（Arbitrum、Base）或專用鏈（dYdX v4、Sei）上，鏈上訂單簿才逐漸具備實用性。

# Order Book vs AMM

| 維度 | Order Book | AMM |
| --- | --- | --- |
| 價格決定 | 買賣雙方掛單 | 數學公式（如 $x \cdot y = k$） |
| 流動性提供者 | 造市商 (Market Maker) | 流動性提供者 (LP) |
| 是否需對手方 | ✓ 需要 | ✗ 直接與資金池交易 |
| 滑價 | 視掛單深度 | 隨交易量公式性增加 |
| 報價精度 | 高（任意價位） | 低（依公式曲線） |
| 適用情境 | 高流動性主流幣對 | 長尾資產、被動 LP |
